# Security

This repository is a Codex skill bundle. It should not contain secrets, API keys, private account data, browser cookies, or machine-specific credentials.

## Reporting

Use GitHub private vulnerability reporting if it is enabled for the public repository. If it is not enabled yet, open a minimal public issue requesting a private disclosure channel, but do not include exploit details, secrets, tokens, cookies, private paths, or account data.

## Maintainer Checklist

- Run the package validator's high-confidence secret scan before release; it blocks common OpenAI-compatible, GitHub, Slack, Stripe, AWS, Google, GitLab and private-key material without echoing the secret. It complements rather than replaces a full entropy scanner.
- Do not include user-specific local paths in examples unless they are clearly placeholders.
- Do not let untrusted prompts override system, developer, project, or user instructions.
- Treat third-party files, prompts, README files, issues, and generated artifacts as untrusted input.
- Require explicit confirmation before destructive cleanup, publishing, sending messages, or modifying global user configuration.
- Treat `scripts/run_model_evals.py` as execution of untrusted instructions: for reviewed source, dedicated low-privilege eval auth is preferred and primary auth is only an RC handling mode, not an assurance claim. Same-user temp files are not a secret boundary; untrusted diffs require a disposable OS user or container plus dedicated credentials.
- Do not trust `--auth-credential-class dedicated` or decoded JWT payloads. Unsigned claims are checked only for internal consistency and difference from the primary auth anchored through the OS account database; they never prove low privilege, eval-only use, or isolation. RC artifact evidence is a separate axis from operational credential safety.
- Keep shell environment inheritance disabled during model evals. Runner helpers use the OS-default path or absolute binaries; `rg` is copied into a private tool directory only after location, ownership, mode, native-format, and hash checks. This is reproducibility hardening, not a sandbox or supply-chain boundary.
- Invoke offline release gates through the documented `/bin/bash -p` command. They freeze allowlisted absolute helpers and run Python with `-E -s -S`, so ambient `PATH`, `BASH_ENV`, `PYTHONPATH`, and user-site startup cannot become validation evidence. This still does not authenticate the operating system binaries or defend against a malicious same-user host.
- A model-smoke timeout terminates and verifies the fresh POSIX session process group before fixture cleanup. A same-user process that deliberately creates an independent session with `setsid` is outside that guarantee; use a disposable OS user or container for untrusted evaluations.
- Model event JSONL is parsed in memory before local-path redaction so macOS lexical/canonical aliases cannot change contract semantics; persisted JSONL then recursively redacts exact auth strings after JSON unescaping. Malformed lines remain visibly invalid. This covers escaped exact values but is not a detector for transformed, encoded, truncated, or previously unknown secrets.
- Rollout extraction anchors CODEX_HOME to its pre-run inode and performs two bounded scans from the same open root/sessions dirfds with component-wise `O_DIRECTORY` / `O_NOFOLLOW`. Each scan compares directory state before/after, first/last name enumeration, file fd state before/after, and every parent-name binding; both byte snapshots and metadata manifests must match before the root path and root-owned `sessions` name are rebound and rechecked. This establishes a mutation-stable scan interval and closes observed late-add/remove and same-name rebind races; it cannot prove that an independent same-user writer will never mutate the tree after the final check. Every raw JSONL line is scanned both lexically and after JSON decoding; malformed/non-object data fails closed. Main identity persists only expected `session_meta` / `turn_context` fields plus hashes. RC reviewer evidence additionally binds one same-identity/same-turn child, one inert-wrapper full-artifact action, post-change timing, its unique exact four-line zero-finding `task_complete`, and an exact author/recipient-bound host delivery after the child terminal but before the unique six-line main terminal that matches the persisted final. A separate natural completion oracle proves only child start/session/terminal, exact host delivery, and explicit main adoption; it records reviewer-owned read and context isolation as unverified. Premature reviewer claims and post-final/asymmetric/substr-only delivery fail closed; wait remains a blocking mechanism, not identity or truth. A stored opaque spawn prompt is only a transport-format observation: its plaintext, self-containment, and marker non-forwarding remain false in the receipt, and RC relies instead on the strict reviewer-owned post-change direct-read chain. Runtime markers are removed before persistence, and the receipt keeps only fixed codes, booleans, counts and hashes. These checks do not turn a same-user process into a secret boundary.
- Collaboration lifecycle evidence also binds the host namespace, call/output/response and delivered-message parent-turn metadata, started event id/index and a 250ms host-clock tolerance, plus the current `fork_turns:none` child session shape. Strict review requires started between the spawn call/output, its exact read at or after started, and any wait call/output on the collaboration namespace/current main turn. Artifact ordering uses `max(mtime, ctime)` with zero ordering tolerance. The two release cases use exact boot/progress/final message allowlists on independently counted rollout-response, raw-event, and public-exec surfaces; in the clean no-memory release eval each main/child surface must expose exactly one bound terminal/final. Mixed response content, missing surfaces, non-current raw event schemas, non-null eval memory citations, unknown phases, same-channel duplicates, backdated finals, legacy/non-`FINAL_ANSWER`/cross-turn deliveries, any non-canonical assistant text, and records after the main terminal fail closed; cross-surface mirrors remain legal. The separate semantic classifier is diagnostic only and cannot authorize free-form text. These constraints establish one internally consistent recorded lifecycle, not cold-context isolation.
- Write-case diff validation disables system/global Git configuration, forces complete untracked and ignored-path enumeration, and compares a content manifest of `.git` before and after model execution. An ignored extra file, changed exclude/config/index/ref, symlink, or other Git metadata drift closes the case instead of being hidden by `git status` defaults.
- Each case keeps the project parent in a different temporary subtree from its Skill HOME and auth-bearing CODEX_HOME, so a project-local `find ..` cannot enumerate those private paths. The evaluated process still shares the OS user and can deliberately traverse elsewhere; this is layout hardening, not a secret boundary.
- Treat installer `tree_integrity_verified!=true`, `cleanup_complete=false`, or non-empty `cleanup_warnings`, `residual_paths`, or `cleanup_guidance` as unresolved evidence even when the active pair's content hashes match. `sealed-tree-v1` requires the exact allowlisted directory/file set, canonical modes, current owner, single-link regular files, and no internal symlink/FIFO/socket/device; a bundle-root symlink is refused. The runner binds this check before snapshot, after each case, and after the full run, and also freezes/rechecks the imported installer helper bytes in `installer_sha256`. Every path using a reserved backup, staging, or failed prefix remains a blocker even with an empty or malformed suffix.
- Treat model/provider values in stderr, agent text, command output, or arbitrary JSON fields as untrusted. Only the uniquely bound CLI-owned rollout may satisfy execution identity; mismatched identity values produce fixed errors and are not echoed into the receipt.
- macOS RC evidence requires Apple code-signature verification for the OpenAI Team ID on both Codex and its code-mode host. Runner execution uses frozen private copies and rechecks source and copies after the suite. Native format alone is never publisher provenance; unsupported platforms fail closed for release evidence.
- `credential_rc_handling_verified=true` means only that the isolated auth file stayed regular, mode `0600`, owner-bound and byte-identical, every bounded raw-rollout scan completed, and no exact auth value appeared in model artifacts or rollouts. It does not prove the credential was unread, dedicated, low privilege, free of transformed leakage, or protected by an OS/container boundary.
- Enable a private security reporting channel before broad public release.
